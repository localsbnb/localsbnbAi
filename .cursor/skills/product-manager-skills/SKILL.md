# Product Manager Skills

## 核心原则

1. **需求先行**：先澄清需求，再进入方案设计
2. **结构化输出**：使用表格、清单等结构化形式，便于评审与追溯
3. **边界清晰**：明确在范围内 / 不在范围内
4. **可验收**：每个功能需求需有可验证的验收标准

---

## PRD 创建流程

### 1. 文档信息（必选）

| 项目 | 说明 |
|------|------|
| 文档名称 | PRD 全称 |
| 产品名称 | 产品/项目名称 |
| 文档版本 | v1.0 |
| 文档状态 | 初稿/评审中/已定稿 |
| 关联文档 | 技术架构、产品方案、工具定义等 |

### 2. 术语与缩写

用表格定义领域术语，避免歧义：

| 术语/缩写 | 全称或含义 |
|-----------|------------|
| XXX | 说明 |

### 3. 产品背景与目标

- **产品背景**：解决的问题、目标用户、使用场景
- **产品目标**：核心目标、业务目标、体验目标
- **目标用户与优先级**：P0/P1/P2 角色、典型场景、核心诉求

### 4. 范围与边界

明确列出：
- **在范围内**：本期要做的能力
- **不在范围内**：本期不做、后续迭代

### 5. 功能需求详述

每个功能模块使用统一表格结构：

| 属性 | 说明 |
|------|------|
| 功能描述 | 一句话说明功能 |
| 优先级 | P0/P1/P2 |
| 输入 | 必填/可选参数及说明 |
| 输出 | 返回数据结构 |
| 业务规则 | 1. 规则1<br>2. 规则2 |
| 验收标准 | 可验证的验收要点 |

### 6. 非功能需求

- 数据脱敏、权限、异常处理、性能等
- 写操作需 Human-in-the-loop 时，明确确认流程

### 7. 附录

- 错误码、话术模板、接口映射等

---

## 需求澄清检查清单

在进入方案设计前，确认：

- [ ] 核心功能点是否明确
- [ ] 目标用户与典型场景是否清晰
- [ ] 业务规则与边界是否定义
- [ ] 优先级（P0/P1/P2）是否排序
- [ ] 验收标准是否可执行

---

## 用户故事格式

```
作为 [角色]，我希望 [能力/动作]，以便 [价值/目标]。
```

**验收条件**（Given-When-Then）：
- Given：前置条件
- When：触发动作
- Then：预期结果

---

## 功能优先级定义

| 优先级 | 含义 | 典型场景 |
|--------|------|----------|
| P0 | 核心必做，MVP 必备 | 主流程、关键能力 |
| P1 | 重要，首版或紧接迭代 | 增强体验、运营支持 |
| P2 | 可选，后续规划 | 锦上添花、长尾需求 |

---

## 输出规范

1. **中文输出**：PRD、需求分析均使用中文
2. **表格优先**：功能描述、参数、规则用表格呈现
3. **分页与截断**：列表类需求明确 page、pageSize、最大范围
4. **敏感数据**：涉及用户隐私时注明脱敏规则
