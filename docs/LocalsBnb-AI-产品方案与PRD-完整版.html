二、产品定位与目标用户

2.1 产品背景

路客云是面向民宿/酒店经营者的多平台 PMS。LocalsBnb AI 将路客云核心能力封装为 MCP，解决「需登录系统才能查数据、办操作」的痛点，使店长、运营、老板等可在任意 AI 终端中通过自然语言完成查询与操作。

2.2 产品定位

LocalsBnb AI 是路客云 PMS 的 AI 化能力出口：通过 token + campId 完成初始化后，可在任何支持 MCP 的 AI 终端（如 Cursor、Claude Desktop、钉钉/飞书 Bot 等）中，以自然语言查询经营数据、待办事项，并在线完成入住/退房、发指引、开关房态、改价等操作。

2.3 产品目标

  • 核心目标：在保证安全与合规的前提下，提供「查 + 办」的完整能力，一次配置、多端复用。
  • 业务目标：提升路客云用户粘性，形成可收费的增值能力（模式 B），为多系统统一 AI 入口打基础。
  • 体验目标：响应稳定、错误提示拟人化、写操作有明确确认步骤，数据输出脱敏。

2.4 目标用户与优先级

优先级角色典型场景核心诉求
P0店长日常查房态、办入住退房、发密码与指引高效、少登系统
P1运营查经营数据、多日报表、房态与房价数据及时、可汇总
P2老板看经营概况、异常与待办，偶尔下发指令一目了然、可授权
P2开发者集成到自建 Bot、自定义工作流协议开放、可配置

2.5 功能优先级定义

优先级含义典型场景
P0核心必做，MVP 必备主流程、关键能力（鉴权、待办、入住退房、房态房价）
P1重要，首版或紧接迭代发指引/密码、多日经营数据、可售情况
P2可选，后续规划渠道查询、用户级模型配置

三、范围与边界

3.1 在范围内

  • 基于 token + campId 的鉴权与初始化
  • 待办（预抵/预离/在住）、订单查询与详情、经营数据（基础/多日）、房态、房价、可售情况等查询能力
  • 订单办理入住/退房、发送入住指引、发送密码；房态开/关；改房价等操作能力（均需 Human-in-the-loop）
  • 数据脱敏、写操作确认流程、异常友好提示、分页与截断策略
  • 模型接入与配置（国内 DeepSeek、海外 GPT）

3.2 不在范围内（本期）

  • 路客云 PC/App 原有功能的改造（仅通过 MCP 暴露能力）
  • 创建订单、取消订单等部分写操作（后续迭代）
  • 多 camp 在同一会话内的切换交互
  • 用户自带 API Key（模式 A）的完整产品化，本期以模式 B 为主

五、产品流程图

5.1 初始化与使用流程

  1. 用户在路客云侧获取 token 与 campId（或由管理员分配）。
  2. 在 AI 客户端（如 Cursor/Claude）的 MCP 配置中填入该 MCP Server 地址及 token、campId。
  3. 用户通过自然语言发起请求；AI 调用 MCP 工具，MCP Server 用 token + campId 鉴权并调用路客云 API。
  4. 若为写操作：AI 先回复「将执行：XXX，是否确认？」→ 用户确认后再执行并返回结果。

六、功能需求详述

6.1 鉴权与初始化

6.1.1 初始化配置

属性说明
功能描述用户或管理员在 AI 客户端配置 MCP 时，需提供 token 与 campId，用于标识身份与数据范围。
优先级P0
输入token（必填）：路客云侧颁发的访问凭证。
campId（必填）：营地/物业 ID，与 token 绑定。
业务规则1. token 与 campId 必须成对有效，否则禁止调用任何工具。
2. 所有 MCP 请求必须携带该 token 与 campId。
3. 权限范围（scope）由路客云根据 token/camp 解析，MCP 层做转发与校验。
验收标准1. 配置正确时，工具可正常调用路客云 API。
2. token 无效或与 campId 不匹配时，返回明确鉴权错误，不调用下游 API。
3. 文档与配置示例中明确说明 token、campId 的获取方式。

6.1.2 权限范围（Scope）

Scope说明涉及工具（示例）
orders:read订单查询query_orders, get_order_details, 待办类查询
orders:write订单写操作check_in_order, check_out_order, send_guide, send_password
rooms:read房态/房价查询query_room_status, query_room_prices, query_availability
rooms:write房态/房价写操作update_room_status, update_room_price, batch_update_prices
finance:read经营/财务数据查询query_operational_data, query_financial_data, analyze_revenue
channels:read渠道查询list_channels, get_channel_details
*全部权限仅管理员或特殊 token

6.2 待办事项（预抵 / 预离 / 在住）

6.2.1 查询预抵订单
属性说明
功能描述查询指定日期范围内「待入住」的订单（预抵），支持分页。
优先级P0
MCP 工具名query_todos_pre_arrival 或复用 query_orders 且 status=pending + 入住日
输入campId（必填）、storeId（必填）、date（必填，YYYY-MM-DD）、page（默认 1）、pageSize（默认 20，最大 100）
输出订单列表：orderId、orderNo、guestName（脱敏）、phone（脱敏）、roomType、checkInDate、checkOutDate、status；total、page、pageSize
业务规则仅返回入住日在查询范围内且状态为待入住；列表敏感字段须脱敏；强制分页，单页最多 100 条
验收标准与路客云「预抵」视图一致；脱敏符合 十、5.2 节
6.2.2 查询预离订单
属性说明
功能描述查询指定日期范围内「待退房」的订单（预离），支持分页。
优先级P0
MCP 工具名query_todos_pre_departure 或 query_orders 且 status=checked_in + 退房日
输入campId/storeId、date（退房日）、page、pageSize
输出同 6.2.1 结构，状态为在住、退房日在查询日
业务规则退房日 = date 且 status=checked_in
验收标准与路客云「预离」一致，脱敏生效
6.2.3 查询在住订单
属性说明
功能描述查询当前在住订单列表，支持分页。
优先级P0
MCP 工具名query_todos_in_house 或 query_orders 且 status=checked_in
输入campId/storeId、page、pageSize
输出在住订单列表，字段同 6.2.1
业务规则仅返回 status=checked_in
验收标准与路客云「在住」一致，脱敏生效

6.3 订单查询

6.3.1 查询订单列表（query_orders）
属性说明
功能描述按条件查询订单列表，支持日期范围、状态、分页等。
优先级P0
输入storeId（必填）、startDate、endDate（必填，YYYY-MM-DD）、status（可选：pending/checked_in/checked_out/cancelled）、roomType、guestName、phone、page（默认 1）、pageSize（默认 20，最大 100）
输出list[]：orderId、orderNo、guestName、phone、roomType、checkInDate、checkOutDate、status、totalAmount、paidAmount；total、page、pageSize；敏感字段脱敏
业务规则日期范围建议不超过 90 天；分页必选，最大 100/页
验收标准与工具定义规范 2.1 一致，且脱敏
6.3.2 获取订单详情（get_order_details）
属性说明
功能描述根据订单 ID 获取订单详细信息。
优先级P0
输入orderId（必填）
输出订单完整信息：orderId、orderNo、guestName、phone、idCard、roomType、roomTypeName、checkInDate、checkOutDate、nights、status、totalAmount、paidAmount、deposit、arrears、channel、remark、createdAt、updatedAt 等；敏感字段脱敏
业务规则仅能查询当前 token/campId 权限范围内的订单；订单不存在或无权限时返回明确错误
验收标准与工具定义规范 2.2 一致，且脱敏

6.4 订单写操作（均需 Human-in-the-loop）

6.4.1 办理入住（check_in_order）
属性说明
功能描述为指定订单办理入住。执行前必须经用户确认。
优先级P0
输入orderId（必填）、roomId（可选，若已排房）
确认流程1. AI 首次调用时 MCP 返回「待确认」结构（操作描述、orderId）。2. 用户确认后 AI 再次调用并传入 confirm=true，MCP 才调用路客云写接口。3. 执行成功后返回订单新状态。
业务规则订单状态必须为待入住；同一订单不可重复入住；操作需落审计日志（谁、何时、哪笔订单）
验收标准未确认不执行；确认后执行结果与路客云一致；审计可查
6.4.2 办理退房（check_out_order）
属性说明
功能描述为指定订单办理退房。执行前必须经用户确认。
优先级P0
输入orderId（必填）
确认流程同 6.4.1
业务规则订单状态必须为在住；不可重复退房；审计落库
验收标准同 6.4.1
6.4.3 发送入住指引（send_guide）
属性说明
功能描述向指定订单的客人发送入住指引（如短信/模板消息）。执行前必须经用户确认。
优先级P1
输入orderId（必填）；可选：templateId、渠道等（以路客云 API 为准）
确认流程同 6.4.1
业务规则依赖路客云现有「发入住指引」能力；若暂无独立接口可标注为后续迭代
验收标准确认后发送成功，且可审计
6.4.4 发送密码（send_password）
属性说明
功能描述向指定订单的客人发送门锁/房门密码。执行前必须经用户确认。
优先级P1
输入orderId（必填）；可选：渠道（以路客云 API 为准）
确认流程同 6.4.1
业务规则依赖路客云现有「发密码」能力
验收标准确认后发送成功，且可审计

6.5 经营数据查询

6.5.1 查询经营数据（query_operational_data）
属性说明
功能描述查询指定日期范围内的经营统计：订单数、收入、入住率、RevPAR 等；支持按日明细。
优先级P0
输入storeId（必填）、startDate、endDate（必填）、type（可选，如 accommodation/voucher，默认 accommodation）
输出totalOrders、totalRevenue、averagePrice、occupancyRate、revpar；dailyData[]：date、orders、revenue、occupancyRate
业务规则日期范围建议不超过 90 天；dailyData 条数可做上限（如最多 31 天），防止 Context 溢出
验收标准与工具定义规范 4.1 一致，且无敏感信息
6.5.2 多日经营数据与汇总

强调「多日」场景；可提供「最近 7 天/30 天」快捷参数。汇总优先返回；明细按需分页或截断。优先级 P1。

6.6 房态查询与写操作

6.6.1 查询房态（query_room_status）
属性说明
功能描述查询指定日期范围内的房态视图（房间维度 + 订单占用）。
优先级P0
输入storeId（必填）、startDate、endDate（必填）、roomTypeId（可选）
输出roomTypes[]：roomTypeId、roomTypeName、rooms[]：roomId、roomName、status（available/occupied/dirty/maintenance）、orders[]（简要）
业务规则日期范围建议不超过 31 天；房间/订单数量大时可分页或仅返回房型级汇总
验收标准与工具定义规范 3.1 一致
6.6.2 开房态 / 关房态（update_room_status）
属性说明
功能描述对指定房源在指定日期进行「开房态」或「关房态」（可售/不可售）。执行前必须经用户确认。
优先级P0
输入storeId（必填）、roomTypeId 或 roomId（必填）、date（必填）、status（必填，如 open/close 或 available/unavailable）
确认流程同 6.4.1
业务规则依赖路客云房态更新接口；审计落库
验收标准确认后房态变更与路客云一致

6.7 房价查询与写操作

6.7.1 查询房价（query_room_prices）
属性说明
功能描述查询指定日期范围内各房型价格及可售数量。
优先级P0
输入storeId、startDate、endDate、roomTypeId（可选）
输出prices[]：date、roomTypeId、roomTypeName、price、available
业务规则日期范围建议不超过 31 天；条数过多时做截断或分页
验收标准与工具定义规范 3.2 一致
6.7.2 查询可售情况（query_availability）

可合并到 query_room_prices 的 available 字段，或独立接口。优先级 P1。

6.7.3 改房价（update_room_price / batch_update_prices）
属性说明
功能描述更新单个或批量房型在指定日期的价格。执行前必须经用户确认。
优先级P0
输入单条：storeId、roomTypeId、date、price。批量：storeId、updates[]：roomTypeId、date、price
确认流程同 6.4.1；批量时需在待确认中列出将改动的条目摘要
业务规则价格需符合路客云规则（如非负、精度）；审计落库
验收标准与工具定义规范 3.3、3.4 一致，且确认后生效

6.8 渠道查询（可选）

6.8.1 list_channels / get_channel_details

获取店铺关联的渠道列表及渠道详情。优先级 P2。输入/输出与《03-工具定义规范》渠道管理工具集一致。验收：只读，有敏感数据则按 十、5.2 脱敏。

七、用户故事（Given-When-Then）

按 product-manager-skills 用户故事格式：作为 [角色]，我希望 [能力/动作]，以便 [价值/目标]。验收条件采用 Given-When-Then。

US-1：店长查今日预抵并办入住

作为店长，我希望通过自然语言查询今日预抵订单并办理入住，以便少登系统、提高效率。

Given已配置 token + campId，MCP 已连接
When用户说「帮我查今天要入住的订单」→ AI 调用待办工具 → 用户选择订单并说「给这个订单办入住」→ AI 展示待确认 → 用户确认
Then返回今日预抵列表（脱敏）；确认后入住成功，订单状态变为在住，审计可查
US-2：运营查近 7 天经营数据

作为运营，我希望查询近 7 天经营汇总与每日明细，以便快速掌握经营情况。

Given已配置 token + campId
When用户说「查一下最近 7 天的经营数据」→ AI 调用经营数据工具
Then返回 totalOrders、totalRevenue、occupancyRate 等汇总；dailyData 含每日明细（最多 31 天截断）；无敏感信息
US-3：老板关某房型明日房态

作为老板，我希望关闭某房型明日可售状态，以便临时控房。

Given已配置 token + campId，有 rooms:write 权限
When用户说「把 301 房型明天关房态」→ AI 调用 update_room_status → 返回待确认 → 用户确认
Then确认后房态变更成功，审计可查

八、产品方案（能力、风控、商业模式）

8.1 核心能力范围

查询能力：当前待办（预抵/预离/在住）、订单详情、基础/多日经营数据、当前房态、当前房价、当前可售。

操作能力（均需 Human-in-the-loop）：订单办理入住/退房、发送入住指引、发送密码；开房态、关房态；改房价。

8.2 约束与风控

  • 数据脱敏：对外输出中姓名、手机号、证件号必须脱敏（MCP/网关层统一处理）。
  • 写操作：所有写操作执行前必须经用户明确确认（Human-in-the-loop）。
  • 异常与容错：超时/限流等场景返回拟人化、可操作的提示语（如「老板，网络有点开小差，稍等一分钟再试」）。
  • 接口设计：分页、截断与摘要优先，避免单次请求打爆 Context 或超时。

8.3 商业模式（以模式 B 为主）

模式 B：路客云统一封装大模型，按次/按月向用户收费（SaaS 思维，体验好）。国内优先接入 DeepSeek，海外优先 GPT；模型由路客云侧配置与路由，用户无需自备 API Key。

模式 A（可选）：用户自带大模型 API Key，适合开发者或高用量客户，作为高级选项。

8.4 模型接入与配置

  • 环境/地域：国内默认 DeepSeek，海外默认 GPT；API Key 由路客云统一管理。
  • 营地级：可为单个 camp 指定模型（如强制 GPT-4），或「自动」沿用地域默认。
  • 用户级：可选，仅在对开发者或高级用户开放时使用。

十、非功能需求

10.1 性能与可用性

需求项说明
响应时间单次工具调用（不含大模型推理）P95 ≤ 5s；路客云 API 超时建议 10–15s，超时后进入 10.4 异常话术。
可用性依赖路客云 API 与 MCP Server 部署可用性；目标与路客云 SLA 对齐。
分页与截断所有列表类接口强制分页，pageSize 最大 100；经营数据 dailyData 等可限制条数（如 31）；单条详情与汇总文本长度可设上限，避免 AI Context 溢出。

10.2 数据脱敏与隐私

需求项说明
脱敏范围所有返回给 AI 客户端的数据中，姓名、手机号、证件号必须脱敏。
脱敏规则姓名：保留姓，其余用 *（如「张**」）。手机号：保留前 3 后 4（如 138****1234）。证件号：仅保留后 4 位或全部 *（可配置）。
实施位置在 MCP Server 或统一网关层，在返回给 AI 之前对结构化结果与文本做脱敏。
合规符合最小必要原则；若后续有「申请查看明文」流程，需单独设计权限与审计。

10.3 写操作与 Human-in-the-loop

需求项说明
适用操作办理入住、办理退房、发送入住指引、发送密码、开/关房态、改房价（含批量）。
流程1）用户表达意图 → 2）AI 调用 MCP 工具 → 3）MCP 返回「待确认」结构 → 4）AI 向用户展示并询问确认 → 5）用户确认后 AI 再次调用并带确认标识 → 6）MCP 调用路客云写接口 → 7）返回执行结果。
确认方式支持自然语言确认（如「确认」「执行」）或客户端按钮；MCP 层通过参数（如 confirm=true）区分「仅预览」与「确认执行」。
审计谁、何时、对哪笔订单/哪间房/哪日房价执行了何操作，在路客云侧落库并可查。

10.4 异常处理与友好提示

场景返回策略
鉴权失败明确提示「未授权」或「token/营地无效」，不暴露内部信息。
路客云 API 超时拟人化话术，如：「老板，现在的网络有点开小差，我没能拿到最新的房态，您稍等一分钟我再试。」
限流「请求有点多，稍等再试。」
部分失败「已拿到 XX，但 YY 暂时没拿到，您可以先看 XX。」
参数错误明确说明缺少或错误的参数，便于用户或 AI 修正。

10.5 模型接入与配置（模式 B）

需求项说明
默认策略国内部署默认 DeepSeek，海外默认 GPT；API Key 由路客云统一管理。
营地级可为单个 camp 指定模型（如 GPT-4o）或「自动」沿用地域默认。
用户级可选，仅在对开发者/高级用户开放时提供。
扩展新增模型时通过配置与适配器扩展，不改业务代码。

十一、数据与接口规范

11.1 分页规范

参数类型默认值最大值说明
pagenumber1—页码，从 1 开始
pageSizenumber20100每页条数

所有列表类响应包含：list、total、page、pageSize。

11.2 日期规范

  • 格式：YYYY-MM-DD。
  • 订单/经营单次查询建议不超过 90 天；房态/房价建议不超过 31 天；超出时可拒绝或截断并提示。

11.3 统一错误响应结构

{
  "code": "非 0 错误码",
  "message": "用户或 AI 可读的提示",
  "friendlyMessage": "可选，拟人化话术，用于直接回复用户"
}

11.4 写操作「待确认」响应结构

{
  "requireConfirmation": true,
  "action": "check_in_order",
  "summary": "为订单 12345 办理入住",
  "params": { "orderId": "12345" },
  "message": "请确认是否执行上述操作。"
}

用户确认后，AI 再次调用时传入上述 params 并增加 confirm: true，MCP 再执行写接口。

十二、验收标准汇总

12.1 按模块

模块验收要点
鉴权与初始化token+campId 正确可调通；错误时返回明确鉴权失败且不调下游。
待办预抵/预离/在住三类查询结果正确、分页与脱敏生效。
订单查询列表与详情与路客云一致，脱敏生效，分页有效。
订单写操作入住/退房/发指引/发密码均需确认后执行，审计可查。
经营数据汇总与多日数据正确，截断策略生效。
房态查询正确；开/关房态需确认后执行，审计可查。
房价查询与可售正确；改价需确认后执行，审计可查。
异常超时/限流等返回友好话术，符合 10.4。
模型配置地域/营地配置生效，国内 DeepSeek、海外 GPT 可切换。

12.2 端到端场景

  1. 店长查今日预抵并办入住：查询今日预抵 → 选择订单 → 办理入住 → 确认 → 执行成功并提示。
  2. 运营查近 7 天经营数据：输入 7 天范围 → 返回汇总与每日明细（或截断提示）→ 无敏感信息。
  3. 老板关某房型明日房态：请求关房态 → 返回待确认 → 确认 → 执行成功并提示。
